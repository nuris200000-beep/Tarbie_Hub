# College Tarbie Hub — MVP спецификация (Desktop-first v1)

## 1) Цель MVP

Запустить первую рабочую версию системы в формате **адаптивного web/desktop приложения**, которая:
- объединяет ключевые процессы воспитательной работы в одном приложении;
- дает заместителю директора базовый контроль и аналитику;
- позволяет психологу, социальному педагогу и куратору работать в едином контуре;
- предоставляет студенту подачу заявок и получение уведомлений.

MVP ориентирован на быстрый запуск и проверку гипотез, без сложных AI-функций и глубокой внешней интеграции на первом этапе. Архитектура сразу закладывается так, чтобы после пилота можно было выпустить мобильную версию без пересборки backend-логики.

## 2) Роли в MVP

Включаем 5 ролей:
- Заместитель директора по воспитательной работе
- Психолог
- Социальный педагог
- Куратор
- Студент

Роль **Родитель** переносится в Phase 2.

## 3) Что входит в MVP (In Scope)

## 3.1 Авторизация и доступ
- Вход по логину/паролю.
- Ролевая модель доступа (RBAC) по 5 ролям.
- Базовое восстановление пароля.

## 3.2 Модуль "Тәрбие ісі" (воспитательная работа)
- Справочник 6 ценностей:
  - Отаншылдық
  - Бірлік
  - Адалдық
  - Талап
  - Құрмет
  - Еңбекқорлық
- Создание и редактирование мероприятий (куратор/админ).
- Привязка мероприятия к ценности, дате, группе, куратору.
- Отметка участия студентов.
- Короткий отчет куратора по мероприятию (текст + фото).
- Просмотр плана/факта по месяцам.

## 3.3 Модуль психолога
- Список студентов и карточка студента.
- Внесение результатов опроса вручную (без автосинхронизации ASPPM на MVP).
- Фиксация уровня риска: низкий / средний / высокий.
- Поле "заключение" и "рекомендация".
- Фильтры по группе и курсу.

## 3.4 Модуль социального педагога
- Социальная карточка студента.
- Категории: многодетный, малообеспеченный, сирота, полусирота, ООП.
- Подача заявок на помощь (студент) с загрузкой документов:
  - `.pdf`, `.jpg`, `.jpeg`, `.png`
- Статусы заявки:
  - Принято
  - На проверке
  - Требуется дополнение
  - Одобрено
  - Отказано
- Комментарий социального педагога при смене статуса.

## 3.5 Dashboard и аналитика (базовая)
- Общий count студентов.
- Count студентов в риске (из модуля психолога).
- Количество мероприятий за период.
- Процент участия по группам (базовый расчет).
- Статистика заявок соцпомощи по статусам.

## 3.6 Уведомления (базовые)
- In-app уведомления:
  - о смене статуса заявки;
  - о новом мероприятии для группы;
  - о необходимости дополнить документы.
- Push можно подключить в конце MVP как optional.

## 3.7 Документы и отчеты
- Экспорт базовых отчетов в PDF:
  - отчет по мероприятиям за месяц;
  - список заявок соцпомощи;
  - список студентов высокого риска.

## 4) Что НЕ входит в MVP (Out of Scope)

- Полноценная интеграция с ASPPM/HR+ API.
- SSO/autologin с внешними системами.
- AI-предикты и сложная интерпретация тестов.
- OCR и авто-проверка документов.
- Роль "Родитель" и отдельный родительский кабинет.
- Excel/Power BI интеграции.
- Двухфакторная авторизация (можно как опция после MVP).
- Автогенерация служебных документов AI-моделями.

## 5) Экранный минимум MVP

- Splash
- Login
- Выбор роли (если у пользователя несколько ролей)
- Главная (Dashboard)
- Список мероприятий
- Создание/редактирование мероприятия
- Детальная карточка мероприятия + участники
- Модуль психолога: список студентов, карточка, ввод результата
- Модуль соцпедагога: список заявок, карточка заявки, загрузка документов
- Уведомления
- Профиль пользователя

## 5.1 Адаптивность интерфейса (обязательно для MVP)

Подход: **Desktop-first**, затем Tablet, затем Mobile.

Breakpoints:
- `>=1280 px` — Desktop (основной режим MVP)
- `768–1279 px` — Tablet
- `<=767 px` — Mobile (базовая поддержка на уровне корректного отображения)

Правила layout:
- Desktop: левый sidebar + верхняя панель + рабочая область с 2-3 колонками.
- Tablet: сворачиваемый sidebar, 1-2 колонки, ключевые карточки в приоритете.
- Mobile: нижняя навигация, 1 колонка, таблицы в карточный вид.
- Минимальная ширина контента: 320 px.
- Все формы и таблицы должны быть usable без горизонтального скролла на `>=768 px`.

Правила компонентов:
- Единая дизайн-система (spacing, typography, buttons, inputs, cards, table, modal).
- Таблицы: для desktop полноценная таблица; для mobile — card-list режим.
- Фильтры и поиск: на desktop inline, на mobile в отдельном bottom sheet/modal.
- Графики: адаптивная высота и упрощение легенд на малых экранах.

## 6) Рекомендуемый стек для MVP (desktop-first)

Для быстрого старта:
- **Frontend (Web/Desktop):** Flutter Web
- **Backend:** Firebase (Auth + Firestore + Storage + Cloud Functions)
- **Analytics:** агрегаты в Firestore + простые графики в приложении
- **Отчеты PDF:** Cloud Function + шаблонный генератор

Почему так:
- минимальная инфраструктура;
- быстрый time-to-market;
- удобно масштабировать после валидации;
- можно переиспользовать UI-слой и бизнес-логику при переходе на iOS/Android.

Переход к mobile после MVP:
- тот же backend и модель данных;
- переиспользование domain/data слоя;
- адаптация экранов под mobile UX (навигация, плотность данных, сценарии офлайн-доступа).

## 7) Минимальная модель данных

- `users` (id, fio, role, groupId, phone, active)
- `groups` (id, name, course, curatorId)
- `students` (id, userId, groupId, socialCategory[], riskLevel)
- `events` (id, title, valueTag, date, groupIds[], curatorId, status)
- `eventAttendance` (id, eventId, studentId, present, note)
- `psychResults` (id, studentId, date, testName, score, riskLevel, conclusion, recommendation)
- `socialApplications` (id, studentId, type, status, createdAt, updatedAt, comment)
- `socialDocuments` (id, applicationId, fileUrl, fileType, verified)
- `notifications` (id, userId, title, body, read, createdAt)

## 8) Безопасность MVP

- RBAC на уровне API/Firestore Rules.
- Шифрование трафика (HTTPS/TLS).
- Логирование действий (кто изменил статус/результат).
- Ограничение доступа к психологическим данным только для роли психолога и замдиректора.
- Ограничение доступа к документам соцпомощи по ролям.

## 9) План реализации (6–8 недель)

**Этап 1 (Неделя 1–2):**
- Проектный каркас, Auth, роли, базовая навигация.
- Основа дизайн-системы и адаптивная сетка (desktop/tablet/mobile).
- Справочники (группы, студенты, ценности).

**Этап 2 (Неделя 3–4):**
- Модуль мероприятий + attendance + отчеты куратора.
- Базовые уведомления.
- Адаптив таблиц/форм под tablet и mobile.

**Этап 3 (Неделя 5–6):**
- Модуль психолога.
- Модуль соцпедагога + загрузка документов.

**Этап 4 (Неделя 7–8):**
- Dashboard метрики.
- PDF-отчеты.
- Адаптивные UX-доработки и кроссбраузерная проверка.
- Тестирование, стабилизация, пилот на 1–2 группах.

## 10) Критерии готовности MVP (Definition of Done)

- 5 ролей могут войти и видеть только разрешенные разделы.
- Куратор может создать мероприятие и отметить посещаемость.
- Психолог может внести результат и выставить риск.
- Студент может отправить заявку и документы.
- Соцпедагог может менять статус и оставлять комментарий.
- Замдиректора видит сводный Dashboard.
- Минимум 3 PDF-отчета экспортируются без ошибок.
- Аудит-лог фиксирует ключевые изменения.
- Интерфейс корректно работает на Desktop и Tablet, а также не ломается на Mobile (`>=320 px`).

## 11) Phase 2 (после MVP)

- Интеграция с ASPPM/HR+ (API/SSO/WebView).
- Нативные/мобильные клиенты (Flutter iOS/Android) на базе существующего backend.
- Родительский кабинет.
- OCR документов и AI-классификация.
- AI-рекомендации по рискам и активности.
- 2FA и расширенная безопасность.
- Расширенная аналитика и прогнозирование.
